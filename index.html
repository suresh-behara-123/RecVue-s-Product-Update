What's New

RecVue Foundation

Usage Hub

We've launched the Usage Hub, a new feature within RecVue's Hub functionality, a powerful addition to the RecVue ecosystem built on our advanced Data Mediation Layer [DML]. It uses a smart data layer to import, organize, and sync usage data from external systems. In Usage Hub, you can set the processing type—such as bulk or regular—depending on the number of transactions you need to handle.

Usage Hub is Different from Usage Hub DML:

The following are the main differences between the Usage Hub and Usage Hub DML:

Functionality Differences:

  • Usage Hub validates and transfers actions into a single processing step, whereas Usage Hub DML involves multiple steps.
  • It supports both regular and bulk processing, whereas Usage Hub DML supports bulk processing.

Interface Differences:

Usage Hub Usage Hub DML
Path: Main menu > Data HUB > Usage Hub Path: Main menu > Data HUB > Usage Hub DML.
It has only two stages: product validation for exception records and usage for success records. It has three stages: validate, transfer, and purge.
It has only two outcomes, namely success and failure. It has outcomes such as business validation, product validation, and transferred.
The features available in buttons such as download, Excel, CSV filter, etc., are consistent on all pages. These options are inconsistent.
It supports all types of user-defined templates from the source functionality. It supports only DML-defined templates.
The KPI shows Total records, success, and Error count. The KPI shows pending, partial, complete, and error count.

What the Usage Hub Does:

The Usage Hub is a dedicated environment for collecting, cleaning, and synchronizing high-volume usage data. It acts as an orchestration layer between your external systems and RecVue, eliminating manual data preparation and ensuring your billing cycle is driven by accurate and validated information.

Key Capabilities:

  • Data Flow Monitoring: Real-time tracking of usage data flows from various external sources.
  • Intelligent Validation: A two-tier check system that verifies both file structure (format) and business logic compliance.
  • Integrated Exception Management: A dedicated workspace to identify, correct, and reprocess failed records without needing to re-upload files.
  • Configure Multiple Sources: Users can configure the source, which determines the type of entity loaded into RecVue.

How the Process Works:

  • Ingestion: Data is gathered from your preferred external sources.
  • Validation: Records undergo strict structural and business-rule audits.
  • Success: Validated data is moved instantly to RecVue.
  • Exception Handling: Failed records are diverted to an Error Table.
  • Reprocessing: Users can correct errors in bulk and resubmit them directly within the Hub.

Business Value:

  • Centralized Visibility: View all usage records—regardless of source—in one unified location.
  • Operational Efficiency: Use bulk-action tools to download, fix, and re-upload erroneous data quickly.

Order Amendment: Update Vendor Sites

We've added the ability to update vendor sites when a vendor has multiple locations. Here's how this feature works in different scenarios.

Amendments - Updating Vendor Sites:

  • Unpaid Payment Lines: Updating the vendor site code will automatically update all schedules. Payments will be generated using the new vendor site code.
  • Partially Paid Payment Lines: Only the remaining unpaid schedules will be updated

    Assumptions:When a Site is updated to a new Site ID from the old one, the amendments engine will not make any adjustments (i.e., create a credit/Debit memo) to the fully paid schedules of the old vendor Site ID.

  • Fully Paid Payment Lines: You cannot update the vendor site code for payment lines that are fully paid. The system will display an error message: "Vendor is fully paid, the vendor site code is not eligible for amendment."

Amendments - Terminating Lines:

When you terminate a payment line after updating the vendor site code, any credit memo schedules will be generated using the new vendor site code.

Payment Adjustments:

When you change the vendor site code and then make adjustments:

  • Adjustments to schedules from the old site will affect the old vendor site code.
  • Adjustments to schedules from the new site will affect the new vendor site code

    Assumption: 1. For manual adjustments against an invoice, any adjustments to previously paid schedules must be recorded against the original vendor Site (i.e., the Vendor Site ID before the update).

Billing Management

RecVue now supports consolidated billing for multiple recurring pricing models, even when they start on different dates. This ensures customers receive one unified invoice per billing cycle, improving clarity and operational efficiency

The Solution:

The new Billing Cycle Offset allows contracts with multiple recurring pricing models that have different start dates to be combined and billed into a single consolidated invoice.

How It Works:

  • Each pricing model is initially billed pro-rated from its individual start date.
  • After the first billing cycle, all models automatically align to a common billing schedule using a "Bill Offset Day"
  • From that point forward, all charges appear on one unified invoice with separate line items for each model.

Example:

  • Model 1 starts April 17, 2025
  • Model 2 starts May 11, 2025
  • First cycle: Each model is billed separately and pro-rated from its start date.
  • Second cycle onward: Both models align to bill together starting May 17, appearing on the same invoice.

Key Benefits:

  • Simplified billing: Customers receive one consolidated invoice instead of multiple separate bills.
  • Accurate charges: Pro-rated amounts are calculated correctly for each model's start date.
  • Consistent billing: After initial alignment, all future invoices follow the same monthly billing schedule.
  • Better customer experience: Clear, unified view of all contract charges.

How to Use It:

Select "Bill Offset Day" as the billing cycle type when setting up your contract. The system will automatically handle pro-rating and alignment.

To align billing for pricing models or contract lines that start on different dates, please update the new billing start date in the “Bill Offset Day” field.

This update should be applied to the model or line item that originally has a later start date than the others, ensuring all charges follow a unified billing schedule going forward.

Revenue Management

Enhancement for SSP Calculation

This release corrects the accuracy of revenue allocation and the Standalone Selling Price (SSP) calculation for cancellation and termination scenarios.

Issues Resolved:

  • Manual Billing Calculations: Cancelled bill schedules were incorrectly included in revenue and SSP totals
  • Negative Amount Cancellations: System now correctly excludes related debit memos from SSP calculation

Feature Updates:

  • Cancellation Handling: Lines marked as Cancellation now have SSP Amount automatically set to zero (applies to both SSP and RSSP methods)
  • Termination Handling: SSP calculations now exclude cancelled schedules from bill schedule totals (applies to both SSP and RSSP methods)
  • No TCV ScenariosWhen Total Contract Value [TCV] is absent for Cancellation or Termination lines, Revenue Amount is calculated from bill schedules excluding Cancellation status (applies to both SSP and RSSP methods).

Strategic Business Value:

  • Enhances accounting flexibility to support evolving business rules and correction scenarios.
  • Strengthens auditability and compliance by maintaining a system-driven and traceable reclassification process.

What's Fixed

Revenue Management

Contract Asset and Contract Liability Schedule Issues

We've resolved two issues affecting Contract Asset (CA) and Contract Liability (CL) schedules:

Issue 1: Missing Scheduled Invoice Amounts in Rollforward Report

Previously, the Rollforward report didn't show scheduled invoice amounts for the current period. This occurred when we optimized the CA/CL scheduling process, which inadvertently disabled the 'Revenue-Allocation Deferred Schedules' profile option.

This is now fixed, and scheduled invoice amounts will show up as expected.

Issue 2: Duplicate Contract Asset Records

Duplicate contract asset records were created for all reversal schedules across the ledger, dating back to the start of each order. This happened because our performance improvements changed how the system fetched previous CA and deferred schedules.

We've fixed it by updating the queries to handle reversal schedules and remove duplicates.

Both issues are now resolved, and the CA/CL scheduling process is working correctly.

Performance Obligation (POB) Change Process Now Recognizes Actual Revenue Recognition Date Updates

The Issue:

When users maintained both an actual release date and a planned release date for a product, changes to the actual date for revenue recognition were not detected by the POB Change and Auto Events process.

The Fix:

The POB Change program now correctly identifies and processes changes to the actual revenue recognition date.

Billing Management

Incorrect Product Line Amount Display

The product page previously displayed incorrect line amounts due to a precision issue, which has now been resolved.

Sorting Issue in Billing Tab

Previously, the billing tab in the new UI [Order UI] didn’t support sorting quantity, invoice number, and so on. This has been fixed, and the sorting functionality is now working as expected.

Overlapping Tier Issue Resolved

Overlapping in tier dates occurred when the price uplift was applied. This issue has been addressed and resolved.

Resolved the issue in the Billing schedule collation:

We've addressed a billing schedule calculation issue that occurred when subscription line start dates fell on the 28th, 29th, 30th, or 31st of the month—particularly in February and leap-year scenarios.

Previously, the system miscalculated billing periods for these edge-case start dates.

Fix: When a subscription line starts on the 28th, billing periods are now calculated correctly. For instance, a line beginning on January 28, 2025, will generate a billing period of February 28, 2025, to March 27, 2025.

We introduced a profile option to control the month-end billing behavior:

  • Disabled (default): System uses existing billing schedule logic.
  • Enabled: System applies the new and required logic for supported 28th-of-month scenarios.

Note: Billing line start dates are 29th, 30th, or 31st of the month—particularly in February and leap-year scenarios, we are working on it and expect it to be fixed in the coming releases.
